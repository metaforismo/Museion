# Museion Demo Video Script and Shot List

Target: **2:40–2:50**, public on YouTube. Recommended format: Francesco on camera for the opening and optional closing; own voice over real product capture for the main body; restrained motion graphics for chapter cards, callouts, and the Codex/GPT-5.6 architecture. Do not replace the working demo with a cinematic launch reel or a long talking-head segment.

## Devpost run of show

This follows the organizer's recommended sequence from Discord and the live submission rules:

1. **This is the project** — name Museion and the learning problem in one sentence.
2. **This is what it does** — show the learner committing before explanation and receiving misconception-aware guidance.
3. **This is how it does that** — show deterministic checking, bounded Maia, independent transfer, evidence, and source-grounded Creator review.
4. **This is how Codex and GPT-5.6 were used** — distinguish the build collaborator from the runtime model roles and the deterministic gates around both.

The video is a walkthrough, not an inventory. Show the smallest set of connected features that proves the product's central claim; use the gallery and README for breadth.

## Recording direction

- Record in 16:9 at 1080p or higher.
- Keep the product on screen for roughly 80% of the runtime.
- Use short animated labels and zooms to direct attention; never fabricate product UI.
- Prefer Francesco's voice for credibility. An AI-assisted voiceover is explicitly allowed if needed.
- Cut typing, loading, cursor travel, and silence. Do not accelerate speech beyond comfortable comprehension.
- Say **Codex** and **GPT-5.6** aloud and explain their distinct roles.

## Script

### 0:00–0:12 — Face-to-camera hook

**Picture:** Francesco, clean background. Museion mark animates in beside the title.

**Voice:** “AI can solve the problem. But if it does the reasoning for us, what did we learn? I built Museion to make thinking visible—and keep the answer outside the model.”

### 0:12–0:30 — Product reveal

**Picture:** Landing page into the dashboard. Animate the seven method labels as a single line: Ground → Predict → Interact → Diagnose → Explain → Transfer → Revisit.

**Voice:** “Museion is a source-grounded learning environment. Every lesson follows seven moves, and the next action comes from the learner's real record—not points, streaks, or an invented readiness score.”

### 0:30–1:05 — Real interaction and diagnosis

**Picture:** Open the Transformation Lab or Recursion Lab. Make one deliberate wrong move. Show the live trace or curve, deterministic feedback, highlighted control, and Maia entering the moment.

**Voice:** “Here I commit to a move before an explanation appears. Deterministic code checks it, recognizes the specific misconception, and marks only the part I should reconsider. Maia sees the activity and asks a smaller question—but she cannot grade the answer or reveal it.”

### 1:05–1:30 — Maia boundary

**Picture:** Show the Maia panel, verified/offline badge, one bounded hint, and the visible statement that answers stay private.

**Voice:** “Every model reply is buffered before delivery. It must pass a typed schema, an allowed-action check, and an answer-leak gate. If it fails, Museion repairs once or falls back to authored guidance. The engine still owns truth.”

### 1:30–1:52 — Independent transfer and revisit

**Picture:** Show the locked transfer state, then Evidence and the Revisit schedule.

**Voice:** “At transfer, Maia leaves the room: one attempt, zero hints, zero solution reveal. Museion records one immediate independent observation, says what that evidence cannot prove, and schedules the concept to return later.”

### 1:52–2:16 — Creator Studio

**Picture:** Source Studio with the two-material Source Pack, provenance roles, normalization hashes, and Course Architect. Briefly show validated review/citation coverage.

**Voice:** “Creators can also bring authorized notes, transcripts, excerpts, or PDFs as one Source Pack. Each material keeps its provenance. Course Architect narrows or refuses thin inputs, and every generated claim must resolve back to supplied evidence before publication.”

### 2:16–2:40 — Codex and GPT-5.6

**Picture:** Simple motion graphic over real Settings/compiler screens: Codex → Luna / Terra / Sol → deterministic gates. Overlay test evidence: 377 passed, browser stress, zero audit findings.

**Voice:** “Codex was my primary engineering collaborator—from architecture and implementation to audits, stress tests, and this release. At runtime, GPT-5.6 Luna extracts source structure, Terra designs learning and powers Maia, and Sol audits and repairs. Their outputs cross deterministic gates they cannot override.”

### 2:40–2:50 — Close

**Picture:** Optional return to Francesco, or hold on the product with repository URL and the line “Try the keyless judge path.”

**Voice:** “Museion does not claim mastery. It creates better conditions for practice—and honest evidence of what a learner can do alone. Try the keyless judge path from the public repository.”
